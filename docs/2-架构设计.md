# 架构设计

## 1. 整体架构

采用分层架构 (Layered Architecture) + 领域驱动设计 (DDD) 思想。

```
┌─────────────────────────────────────────────────────────────┐
│                     Presentation Layer                       │
│                      (FastAPI Routes)                        │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                      Application Layer                        │
│                    (Services + Schemas)                       │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                        Domain Layer                           │
│                       (Models + Entities)                     │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    Infrastructure Layer                      │
│              (Repositories + External Services)               │
└─────────────────────────────────────────────────────────────┘
```

## 2. 后端架构

### 2.1 核心模块

| 模块 | 职责 |
|------|------|
| `api/v1` | API路由定义，请求处理 |
| `core` | 核心配置、安全、数据库连接 |
| `models` | SQLAlchemy ORM 模型 |
| `schemas` | Pydantic 数据验证模式 |
| `services` | 业务逻辑层 |
| `repositories` | 数据访问层 |

### 2.2 设计模式

- **Repository Pattern**: 数据访问抽象
- **Service Layer**: 业务逻辑封装
- **Dependency Injection**: FastAPI 依赖注入
- **Factory Pattern**: 模型策略工厂

### 2.3 配置管理

```python
# 环境变量配置优先级
# 1. 环境变量 (.env)
# 2. docker-compose.yml 环境配置
# 3. config.py 默认值
```

## 3. 前端架构

### 3.1 目录结构

```
frontend/src/
├── components/       # 组件
│   ├── ui/          # 基础UI组件 (shadcn/ui)
│   ├── layout/      # 布局组件
│   ├── project/     # 项目相关组件
│   ├── episode/      # 章节相关组件
│   └── ...
├── pages/           # 页面
├── hooks/           # 自定义Hooks
├── stores/          # 状态管理 (Zustand)
├── api/             # API客户端
└── utils/           # 工具函数
```

### 3.2 状态管理

使用 Zustand 进行状态管理：

| Store | 职责 |
|-------|------|
| `useProjectStore` | 项目状态 |
| `useEpisodeStore` | 章节状态 |
| `useModelStore` | 模型配置 |
| `useUIStore` | UI状态 |

### 3.3 组件设计原则

- **原子设计**: atom → molecule → organism → template → page
- **受控组件**: 表单组件完全受控
- **组合优于继承**: 使用组合模式复用逻辑
- **单一职责**: 每个组件只做一件事

## 4. 数据库架构

### 4.1 PostgreSQL 配置

```yaml
# docker-compose.yml
postgres:
  image: postgres:15
  environment:
    POSTGRES_DB: shortplay
    POSTGRES_USER: shortplay
    POSTGRES_PASSWORD: ${DB_PASSWORD}
  volumes:
    - postgres_data:/var/lib/postgresql/data
```

### 4.2 索引策略

- B-tree: 等值查询、范围查询
- GIN: JSON字段、全文搜索
- 向量索引: PostgreSQL pgvector

### 4.3 分表策略

| 数据量级 | 策略 |
|---------|------|
| < 100万 | 单表 |
| 100-1000万 | 按项目分表 |
| > 1000万 | 分库分表 |

## 5. 缓存架构

### 5.1 Redis 用途

| 用途 | 数据结构 | 过期时间 |
|------|----------|----------|
| 会话管理 | String | 24h |
| 模型缓存 | Hash | 1h |
| 项目状态 | String | 5min |
| WebSocket映射 | Hash | Session生命周期 |

### 5.2 缓存策略

- **Cache-Aside**: 应用直写数据库，缓存失效时更新
- **Write-Through**: 写缓存时同步写数据库
- **TTL**: 设置合理过期时间

## 6. 消息队列架构

### 6.1 RabbitMQ 交换机

| 交换机 | 类型 | 用途 |

| `shortplay.direct` | Direct | 任务分发 |

### 6.2 队列设计

| 队列 | 消费者 | 用途 |
|------|--------|------|
| `video.generate` | VideoWorker | 视频生成 |
| `image.generate` | ImageWorker | 图片生成 |
| `extract.info` | ExtractWorker | 信息提取 |

## 7. API 架构

### 7.1 版本策略

- 当前版本: `v1`
- URL格式: `/api/v1/{resource}`
- 向后兼容: 不破坏性修改

### 7.2 认证策略

- JWT Bearer Token
- Token有效期: 24h
- Refresh Token: 7d

### 7.3 限流策略

| 级别 | 限制 |
|------|------|
| 全局 | 1000 req/min |
| 登录接口 | 10 req/min |
| 生成接口 | 5 req/min |

## 8. 安全架构

### 8.1 防护措施

- SQL注入: ORM参数化查询
- XSS: 输入输出编码
- CSRF: JWT Token验证
- CORS: 域名白名单

### 8.2 敏感数据

- 密码: bcrypt 哈希
- API Key: AES加密存储
- 日志: 脱敏处理
